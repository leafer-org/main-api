# План развития маркетплейса

## Что уже сделано

| Фича | Что делает | Статус |
|------|-----------|--------|
| **IDP** | Авторизация, пользователи, роли, сессии | Готово |
| **Media** | Загрузка и хранение файлов (S3) | Готово |
| **CMS** | Каталог: категории, типы товаров, атрибуты | Готово |
| **Organization** | Организации, сотрудники, роли, подписки, товары (управление) | Готово |
| **Discovery** | Лента, поиск, рекомендации, лайки, фильтрация (только списки) | Готово |

Вся базовая инфраструктура: Kafka, Outbox, транзакции, E2E тесты, OpenAPI-контракты.

### Ключевой разрыв: просмотр товара

Сейчас пользователь может найти товар (Discovery) и организация может его создать (Organization).
Но **открыть товар и взаимодействовать с ним — нельзя:**

- Нет публичного `GET /items/:id` — детальной страницы товара для обычного пользователя
- Discovery отдаёт только карточки в списках (`ItemListView`: title, price, rating, owner)
- Organization `GET /items/:id` — только для сотрудников (управление черновиком)
- В SPEC.md Discovery прямо сказано: "детальная страница товара — другой сервис"
- Нет логики взаимодействия с виджетами (бронирование, оплата зависят от виджетов)

## Что делать дальше

### Волна 1 — Без этого нельзя запускаться

#### 1. Тикет-система с LLM-модерацией

**Зачем:** Блокер номер один. Товары и организации отправляют `moderation-requested` в Kafka, handlers для `approve`/`reject` в Organization написаны. Но ответить некому — товары застрянут в "на модерации" навсегда.

**Решение:** Конфигурируемая тикет-система (как Jira) со спецификой бизнес-событий. Триггеры (в коде) создают тикеты в настраиваемых списках (из админки). LLM-автоматизация на Claude модерирует текст + картинки. Ручной фолбек для спорных случаев.

**Детальный план:** [plans/ticket-system.md](plans/ticket-system.md)

**Объём:** Фаза 1 (работающая модерация) — средний. Фазы 2-4 (админка, орг-контекст, расширения) — итеративно.

---

#### 2. Просмотр товара (Item Showcase)

**Зачем:** Это недостающее звено между "нашёл в ленте" и "записался/купил". Без него маркетплейс — каталог карточек без возможности открыть товар.

**Суть:** Новая фича, которая:
- Отдаёт полные данные товара с развёрнутыми виджетами для публичного пользователя
- Определяет `toDetailView()` — как каждый виджет рендерится на странице товара
- Регистрирует просмотр (`interaction.recorded`, type: `view`) для рекомендаций
- Показывает доступные действия в зависимости от виджетов (есть schedule → кнопка "записаться", есть payment → цена и кнопка "купить")

**Варианты реализации:**

A) **Отдельная фича `item-showcase`** (как сказано в SPEC — "другой сервис"):
  - Читает данные из тех же проекций Discovery (или своих)
  - Слушает `item.published` / `item.unpublished` через Kafka
  - Свой контроллер `GET /items/:id`
  - Определяет detail view и доступные действия

B) **Расширение Discovery** (проще, данные уже есть):
  - Добавить `GET /items/:id` в Discovery
  - Добавить `toDetailView()` маппер
  - Разделение на list view / detail view уже заложено в SPEC

**Рекомендация:** Вариант B для MVP, вариант A — если showcase будет обрастать сложной логикой.

**Что нужно определить:**
- Какие данные виджета нужны для detail view vs list view
- Какие действия доступны для каждого типа виджета
- Нужна ли авторизация (анонимный просмотр или только для залогиненных)

**Объём:** Средний.

---

#### 3. Отзывы (Reviews)

**Зачем:** Доверие — основа маркетплейса. Всё уже заложено: kernel-события `review.created` / `review.deleted`, типы `ReviewTarget` (item | organization), Discovery виджеты `item-review` и `owner-review`, projection handler уже работает. Осталось написать сам агрегат.

**Зависимость:** Нужен Item Showcase (#2), чтобы отзывы было где показывать.

**Что нужно:**
- Агрегат `Review` (создание, удаление, возможно редактирование)
- Привязка к `Item` или `Organization` (тип `ReviewTarget` уже в kernel)
- Подсчёт среднего рейтинга и количества (уже в событиях: `newRating`, `newReviewCount`)
- Ограничение: один отзыв на пользователя на сущность
- Kafka-публикация → Discovery обновляет рейтинги в виджетах

**Объём:** Средний. Контракты готовы.

---

#### 4. Бронирование / Запись (Booking)

**Зачем:** Core-транзакция маркетплейса услуг. Без этого маркетплейс — красивый каталог.

**Зависимость от виджетов:** Бронирование привязано к конкретным виджетам товара:
- `schedule` (расписание) → запись на слот в расписании
- `event-date-time` (даты) → запись на конкретную дату
- `payment` (оплата) → определяет нужна ли оплата при бронировании
- Товар без этих виджетов — не бронируемый (только просмотр)

**Зависимость:** Нужен Item Showcase (#2), чтобы пользователь мог видеть слоты и записываться.

**Что нужно:**
- Агрегат `Booking` (создание, подтверждение, отмена, завершение организацией)
- Статусы: `pending` → `confirmed` → `completed` / `cancelled`
- Проверка доступности слотов (расписание / даты)
- API для пользователя: создать запись, мои записи, отменить
- API для организации: список записей, подтвердить, завершить
- Kafka-событие `interaction.recorded` (type: booking) → Gorse для рекомендаций
- Уведомление организации о новой записи (пока можно без push — просто endpoint "новые записи")

**Объём:** Большой.

### Волна 2 — Нужно для роста

#### 5. Уведомления (Notifications)

**Зачем:** Пользователи и организации должны узнавать о: новая запись, результат модерации, отзыв, статус бронирования. Без уведомлений engagement падает.

**Что нужно:**
- Kafka consumers на бизнес-события → формирование уведомлений
- Каналы: push (Firebase), in-app (для начала)
- REST API: список уведомлений, mark as read
- Замена `MockOtpSender` на реальный SMS-провайдер (в рамках этой фичи)

**Объём:** Средний-большой. Начать с in-app + push.

---

#### 6. Баннеры (Banners)

**Зачем:** Монетизация + продвижение. OpenAPI-контракт уже написан (`banners.yaml`), схема определена.

**Что нужно:**
- CRUD для баннеров (админка)
- Отображение на главной / в категориях
- Привязка к периоду показа

**Объём:** Маленький. Контракт готов.

---

#### 7. Чат / Сообщения

**Зачем:** Пользователь хочет уточнить детали перед записью. Организация хочет связаться с клиентом.

**Что нужно:**
- Агрегат `Conversation` (создание, сообщение, прочитано)
- Участники: User <-> Organization
- WebSocket или SSE для real-time
- REST API: диалоги, история, отправка

**Объём:** Большой. Можно отложить если есть телефоны организаций.

---

### Волна 3 — Масштабирование и монетизация

#### 8. Платежи и подписки

**Зачем:** Subscription plans (`free`, `individual`, `team`) в коде есть, но смена плана — просто команда. Нужна интеграция с платёжной системой.

**Что нужно:**
- Платёжный провайдер (ЮKassa / Stripe)
- Оплата подписок: продление, отмена
- Webhook-обработчик
- Downgrade при неоплате (логика `DowngradeToFree` уже есть)
- Опционально: оплата за бронирование (если услуги платные)

**Объём:** Большой.

---

#### 9. Аналитика для организаций

**Зачем:** Организации хотят видеть: просмотры, записи, конверсия, популярные товары.

**Что нужно:**
- Агрегация `interaction.recorded` (view, click, booking) по организации/товару
- Дашборд и REST API

**Объём:** Средний.

---

#### 10. Жалобы и модерация контента

**Зачем:** Пользователи должны жаловаться на товар, отзыв, организацию.

**Что нужно:**
- Создание жалобы (reason, target)
- Очередь для модератора
- Действия: предупреждение, скрытие, бан

**Объём:** Средний.

## Рекомендуемый порядок

```
MVP (запуск):
  1. Модерация (авто-approve)       — чтобы товары публиковались
  2. Item Showcase (просмотр товара) — чтобы товар можно было открыть
  3. Отзывы                          — доверие и рейтинги
  4. Бронирование                    — core-транзакция

Рост (первые пользователи):
  5. Уведомления (push + in-app)
  6. Баннеры
  7. Чат

Масштаб:
  8. Платежи
  9. Аналитика
  10. Жалобы и модерация
```

## Граф зависимостей

```
Модерация ──────────────┐
                        ├──> Товары публикуются
Item Showcase ──────────┤
  │                     │
  ├── Отзывы            │    (нужен showcase чтобы показывать)
  ├── Бронирование      │    (нужен showcase + виджеты schedule/event-date-time)
  │     │
  │     └── Платежи          (оплата бронирования)
  │
  └── interaction.view ──> Gorse рекомендации улучшаются

Бронирование ───┐
Модерация ──────┼──> Уведомления  (события для нотификаций)
Отзывы ─────────┘

Уведомления ──> Реальный OTP sender (SMS)
```
